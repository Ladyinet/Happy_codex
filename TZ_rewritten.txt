# Технічне завдання
## Python-бот для алгоритмічної торгівлі на BingX
### На основі логіки TradingView-стратегії “C - limit 14 SHORT (EVEN) bingX COMP”

---

## 1. Мета проєкту

Потрібно створити Python-бота, який максимально точно відтворює логіку поточного Pine Script для short-стратегії з DCA, повним TP, trailing TP, sub-cover, обмеженням частоти ордерів, live-sync стартом, журналюванням подій та Telegram-сповіщеннями.

Бот має працювати з **BingX Perpetual Futures** через API та підтримувати окремі режими:
- **backtest / simulation** — перевірка логіки на історичних або модельованих даних;
- **dry_run** — робота на реальному ринку без реального виставлення ордерів;
- **live** — реальна торгівля через BingX API.

Головна мета — забезпечити безпечний шлях від перевірки логіки до реального запуску на коштах без втрати ключових правил Pine-стратегії.

---

## 2. Загальна концепція стратегії

Бот реалізує алгоритмічну **short-стратегію з усередненням вгору (DCA)**, яка:
- відкриває початковий short;
- додає DCA-ордери при русі ціни проти позиції;
- виконує часткове закриття останнього лота (**sub-cover**) за логікою LIFO;
- виконує повне закриття всієї позиції (**Full TP**);
- підтримує **trailing TP** після дотику TP;
- обмежує частоту ордерів за бар і за rolling-window;
- веде повний локальний стан, журнал подій і Telegram-сповіщення.

## 4. Режими роботи

Усі терміни в проєкті мають бути уніфіковані. Дозволені лише такі назви режимів:

### 4.1 `backtest`
Історичне тестування на наборі свічок.

Особливості:
- використовується історичний OHLCV;
- виконання моделюється локально;
- ордери на біржу не надсилаються;
- результати записуються в журнал;
- використовується для звірки логіки з Pine Script.

### 4.2 `dry_run`
Робота на **реальному потоці ринкових даних**, але **без реального виставлення ордерів**.

Особливості:
- використовуються реальні котирування BingX;
- сигнали рахуються в реальному часі;
- ордери не відправляються на біржу;
- ведеться локальна віртуальна позиція, лоти, PnL, журнали;
- Telegram повідомляє про “віртуальні” відкриття/закриття;
- режим обов’язковий перед переходом у live.

### 4.3 `live`
Реальна торгівля через BingX API.

Особливості:
- ринкові дані отримуються з біржі;
- ордери відправляються на біржу;
- локальний state синхронізується з BingX;
- виконується перевірка fills;
- при розсинхронізації застосовується політика safe-stop або manual confirm.

### 4.4 Вимоги до однаковості логіки
Логіка стратегії у режимах `dry_run` і `live` має бути однаковою. Відрізнятися може лише шар виконання ордерів.

## 5. Модель виконання стратегії

### 5.1 Базова модель
Стратегія працює **по закриттю бару**.

**Execution model:**
- `close-only execution`
- базова модель виконання: `fill_price = close`

### 5.2 Джерело тригера
Умови тригерів можуть визначатися залежно від конфігурації:
- по `high/low`;
- або по `open/close body`.

### 5.3 Формалізація `touch`
Потрібно підтримати чіткий параметр конфігурації, який визначає, що вважати дотиком до рівня:
- `TOUCH_MODE=wick` — дотик по high/low;
- `TOUCH_MODE=body` — дотик по open/close body;
- `TOUCH_MODE=close` — рівень вважається досягнутим лише на close.

### 5.4 Пріоритети подій на одному барі
Щоб уникнути неоднозначностей, події на одному барі обробляються в такому порядку:
1. Перевірка критичних умов stop/safe-stop.
2. Перевірка Full TP.
3. Перевірка trailing TP exit.
4. Перевірка sub-cover.
5. Перевірка DCA.
6. Логування та оновлення state.

### 5.5 Правило конфлікту Full TP vs DCA
Якщо на одному барі одночасно виконуються умови **Full TP** і **DCA**, пріоритет завжди має **Full TP**.

### 5.6 Правило конфлікту Full TP vs sub-cover
Якщо на одному барі одночасно виконуються умови **Full TP** і **sub-cover**, пріоритет має **Full TP**, якщо інше явно не визначено окремим параметром конфігурації.

## 6. Параметри стратегії

### 6.1 Базовий ордер
Потрібно підтримати два режими розрахунку стартового short-ордера:

1. **Фіксований розмір у coin**
   - `FIRST_SELL_QTY_COIN`

2. **Відсоток від фіксованого equity у USDT**
   - `USE_EQUITY_PCT_BASE=true`
   - `BASE_ORDER_PCT_EQ`
   - `EQUITY_FOR_SIZING_USDT`

### 6.2 Full TP
- `TP_PERCENT = 1.1`

Розрахунок:
```python
full_tp_price = avg_price * (1 - TP_PERCENT / 100)
```

### 6.3 Trailing TP
- `CALLBACK_PERCENT = 0.2`

Алгоритм:
1. ціна торкається Full TP;
2. активується trailing;
3. фіксується мінімум після активації;
4. розраховується рівень виходу:

```python
trail_stop = trailing_min * (1 + CALLBACK_PERCENT / 100)
```

5. позиція закривається, якщо:
```python
close >= trail_stop
```

### 6.4 Sub-cover TP
- `SUB_SELL_TP_PERCENT = 1.3`

Розрахунок для останнього лота:
```python
last_lot_tp = last_lot.entry_price * (1 - SUB_SELL_TP_PERCENT / 100)
```

### 6.5 Максимум DCA
- `MARGIN_CALL_LIMIT = 244`

## 7. DCA логіка

### 7.1 Загальний принцип
Стратегія відкриває нові short-ордери при рості ціни проти поточної позиції.

### 7.2 Наступний рівень DCA
```python
next_level_price = last_fill_price * (1 + rise_percent / 100)
```

### 7.3 Нелінійні early-level кроки
- `lvl2 = 0.3%`
- `lvl3 = 0.4%`
- `lvl4 = 0.6%`
- `lvl5 = 0.8%`
- `lvl6 = 0.8%`
- `after = LINEAR_RISE_PERCENT`

### 7.4 Множники DCA-ордерів
- `lvl2 = 1.5x`
- `lvl3 = 1.0x`
- `lvl4 = 2.0x`
- `lvl5 = 3.5x`
- `after = 1.0x`

### 7.5 Захист від DCA після TP touch
Потрібно підтримати параметр:
- `BLOCK_DCA_ON_TP_TOUCH=true/false`

Логіка:
- якщо `true`, то після дотику TP DCA до підтвердження TP не відкривається;
- якщо `false`, поведінка визначається базовою логікою стратегії.

## 9. Sub-cover логіка

### 9.1 Умова активації
Sub-cover може працювати лише після того, як:
```python
num_sells > 5
```

### 9.2 Що закривається
Закривається **останній лот** за правилом LIFO.

### 9.3 Режими підтвердження
Потрібно підтримати параметр:
- `SUBCOVER_CONFIRM_MODE=off`
- `SUBCOVER_CONFIRM_MODE=breakeven`
- `SUBCOVER_CONFIRM_MODE=subcover_tp`

### 9.4 Базова логіка
```python
if low <= last_lot_tp:
    close_last_lot()
```

### 9.5 Поведінка при конфлікті з Full TP
Якщо на одному барі одночасно активуються `sub-cover` і `full_tp`, за замовчуванням виконується **Full TP**.

## 10. Перерахунок позиції

Після кожного нового fill:

```python
pos_size_abs += qty
pos_proceeds_usdt += qty * price
avg_price = pos_proceeds_usdt / pos_size_abs
```

Після часткового закриття:
- потрібно коректно перерахувати `pos_size_abs`;
- зберегти історію часткових закриттів;
- оновити LIFO-структуру лотів;
- оновити `num_sells`, якщо це частина визначеної логіки циклу.

## 11. Full TP логіка

При старті live-режиму бот повинен:

## 15. Джерело істини та синхронізація state

### 15.1 Проблема розсинхронізації
Якщо локальний стан і стан на BingX розходяться, бот не повинен мовчки продовжувати торгівлю.

### 15.2 Базове правило
У live-режимі джерелом істини є комбінація:
1. локального state storage;
2. журналу подій та fills;
3. поточного стану біржі.

### 15.3 Політика при розсинхронізації
Потрібно підтримати параметр:
- `DESYNC_POLICY=safe_stop`
- `DESYNC_POLICY=rebuild_from_exchange`
- `DESYNC_POLICY=manual_confirm`

### 15.4 Рекомендоване значення для першої версії
- `DESYNC_POLICY=safe_stop`

### 15.5 Safe-stop
При розсинхронізації бот:
- блокує нові ордери;
- логгує помилку рівня CRITICAL;
- надсилає Telegram alert;
- очікує ручного підтвердження або окремої команди sync.

## 16. Відновлення після рестарту

### 16.1 Вимога
Бот повинен вміти відновити роботу після рестарту без втрати внутрішньої логіки.

### 16.2 Що потрібно зберігати локально
- всі відкриті лоти;
- історію відкриття та часткових закриттів;
- order_id / client_order_id;
- avg price;
- pos size;
- proceeds;
- num_sells;
- last_fill_price;

- статистику обмежень по барах і rolling window;

### 16.3 Вимога до storage
Біржа не є достатнім джерелом для відновлення LIFO-лотів. Тому бот зобов’язаний вести **власне локальне сховище state**.

### 16.4 Формат storage
Рекомендовано:
- **SQLite** як основний варіант;
- JSON — лише для простого debug або резервного snapshot.

## 19. Політика помилок API

### 19.1 Обов’язкові класи обробки
Потрібно розділяти:
- мережеві помилки;
- timeout;
- rate limit;
- помилки валідації ордера;
- невідомий статус ордера;
- partial fill;
- критичні помилки синхронізації.

### 19.2 Retry policy
Потрібно підтримати:
- `API_MAX_RETRIES`
- `API_BACKOFF_BASE_SECONDS`
- `API_BACKOFF_MODE=exponential`

Рекомендовані значення:
- `API_MAX_RETRIES=3`
- `API_BACKOFF_BASE_SECONDS=1`
- `API_BACKOFF_MODE=exponential`

### 19.3 Часткове виконання
При `partial fill` бот зобов’язаний:
- оновити локальний розмір позиції;
- оновити лоти;
- записати проміжний статус ордера;
- не дублювати ордер повторно без перевірки біржі.

### 19.4 Невідомий статус ордера
Якщо відповідь API не підтверджує остаточний стан ордера:
1. бот перевіряє статус ордера через окремий запит;
2. звіряє fills;
3. лише після цього приймає рішення про повтор.

### 19.5 Safe-stop
При критичних помилках синхронізації або сумнівному стані ордера бот переходить у safe-stop.

## 24. Telegram інтеграція

### 24.1 Загальна вимога
Усі торгові події мають:
- записуватися у локальний журнал;
- дублюватися в Telegram.

### 24.2 Підписка користувачів
Будь-який користувач може:
1. знайти бота;
2. натиснути `/start`;
3. бути доданим до таблиці підписників.

### 24.3 Таблиця підписників
```text
subscribers
- id
- chat_id
- username
- first_name
- created_at
- is_active
```

### 24.4 Джерело даних для команд
Команди `/status`, `/position`, `/pnl` повинні використовувати:
1. локальний runtime state;
2. локальний storage;
3. а не прямий виклик біржі на кожен запит.

Біржа використовується для синхронізації, а не як джерело відповіді на кожну команду.

### 24.5 Мінімальний набір команд
- `/start` — підписка
- `/stop` — відписка
- `/status` — стан бота
- `/position` — поточна локальна позиція
- `/pnl` — поточний локальний PnL
- `/sync` — ручна звірка з біржею

### 24.6 Типи повідомлень
- `BOT_STARTED`
- `FIRST_SHORT`
- `DCA_SHORT`
- `SUB_COVER`
- `FULL_COVER`
- `TRAILING_TP`
- `RESET_CYCLE`
- `SAFE_STOP`
- `ERROR`

### 24.7 Структура повідомлення
Кожне повідомлення повинно містити:
- `timestamp`
- `mode`
- `symbol`
- `event_type`
- `price`
- `qty`
- `position_size`
- `avg_price`
- `pnl`
- `reason`
- `cycle_id`

### 24.8 Throttling
- `TELEGRAM_MAX_MESSAGES_PER_SECOND = 10`

### 24.9 Fail-safe
Telegram-модуль не повинен зупиняти бота при:
- Telegram API error;
- timeout;
- network error.

## 28. Критерії приймання

Рішення вважається прийнятним, якщо:

1. Бот відтворює логіку Pine Script без втрати ключових правил стратегії.
2. Є три робочі режими: `backtest`, `dry_run`, `live`.
3. `dry_run` працює на реальному ринку без відправлення реальних ордерів.
4. `live` виконує реальні ордери через BingX API.
5. Підтримується LIFO-логіка лотів.
6. Підтримується відновлення state після рестарту.
7. Є чітка політика safe-stop при розсинхронізації або критичних помилках.
8. Є коректна нормалізація цін і кількостей під правила BingX.
9. Кожен ордер і кожна подія логуються локально.
10. Кожен ордер і ключові події дублюються в Telegram.
11. Команди Telegram працюють із локального state/storage.
12. Усі параметри задокументовані в `.env`.
