Мета проєкту
Потрібно створити Python-бота, який відтворює логіку поточного Pine Script: шорт-стратегію з DCA, контрольним TP, суб-ковером, обмеженням частоти ордерів, live-sync стартом і журналюванням подій. Бот має працювати з BingX через API та підтримувати окремі режими для тестового прогону і реальної торгівлі, щоб можна було безпечно перевіряти поведінку стратегії перед запуском на реальні кошти.

Функціональні вимоги
Реалізувати шорт-стратегію з початковим входом, DCA-добавками, повним TP та суб-ковером останніх лотів.

Підтримати LIFO-структуру лотів, щоб закривати саме останній відкритий лот при sub-cover логіці.

Додати два режими роботи:

Trading / Simulation — ордери не відправляються на біржу, а моделюються локально.

Real Trading — ордери відправляються на BingX через API.

Кожен ордер має:

записуватись у локальний журнал;

дублюватись у Telegram-боті FatherBot;

бути доступним для всіх, хто запустив бота командою /start.

Реалізувати захист від надмірної частоти ордерів: ліміт на ордери за бар і за 3-хвилинне вікно.

Реалізувати live-sync старт, щоб бот коректно починав роботу з конкретного моменту часу та скидав стан при запуску live-режиму.

Логіка стратегії
Вхід у позицію
Початковий шорт відкривається, коли виконані умови старту циклу та бот не перебуває в reset-стані. Розмір першого ордера має підтримувати два варіанти розрахунку: фіксований розмір у coin або відсоток від заданого equity у USDT.

DCA
При русі ціни проти позиції бот відкриває наступні шорти на рівнях, що визначаються нелінійними кроками для ранніх рівнів і лінійною логікою для пізніших. Для кожного наступного рівня застосовується множник до базового розміру ордера, який змінюється залежно від номера рівня.

Full TP
Потрібно реалізувати повне закриття всієї позиції за цільовим TP, а також trailing-варіант повного TP, якщо активовано callback-логіку. Повне закриття може вимагати підтвердження закриття ціни нижче TP, якщо це ввімкнено параметром конфігурації.

Sub-cover
Після накопичення достатньої кількості шортів бот має вміти закривати останній лот окремо, якщо він досяг свого локального TP. Для цього треба підтримати режим підтвердження закриття:

off;

breakeven;

subcovertp.

Скасування та пріоритети
Потрібно зберегти пріоритет TP над DCA, тобто при досягненні умов повного TP DCA не повинен відкриватися. Якщо ціна торкається TP, але повне підтвердження ще не настало, поведінка залежить від налаштування blockDcaOnTpTouch.

Архітектура Python-проєкту 
Модулі

strategy/core.py — реалізація торгової логіки, стану позиції та циклів.

strategy/risk.py — ліміти, throttle, перевірка дозволу на ордер.

runner/live.py — реальний трейдинг.

main.py — запуск через CLI або env-параметр.

Стан, який треба зберігати
поточний розмір позиції;

середня ціна входу;

кількість відкритих шортів;

список лотів у порядку відкриття;

остання ціна входу;

наступний рівень DCA;

прапорець trailing;

прапорець resetCycle;

статистика по ордерах за бар і за 3 хвилини.

Режими роботи
Симуляція
У цьому режимі бот використовує ті ж правила, що й live, але:

не відправляє ордери на BingX;

записує всі дії в локальний лог;

відтворює execution-логіку максимально близько до реального режиму.

Реальний трейдинг
У цьому режимі бот:

читає сигнали;

перевіряє ризик-умови;

відправляє ордери на BingX;

після підтвердження виконання оновлює внутрішній стан;

негайно дублює кожен ордер у Telegram.

Telegram-інтеграція
Потрібно реалізувати Telegram-канал/бот-повідомлення так, щоб:

існував бот, створений FatherBot;

будь-який користувач може написати /start і отримувати результати;

кожен ордер надсилається у вигляді повідомлення з типом дії, символом, ціною, кількістю, часом і причиною входу/виходу.

Рекомендований формат повідомлення:

тип ордера;

символ;

side;

qty;

fill price;

reason;

mode;

timestamp;

поточний стан позиції.

Файл .env
У .env потрібно зберігати всі секрети та параметри запуску, а в коментарях біля кожного параметра чітко описати його призначення. Наприклад:

BINGX_API_KEY — API key для BingX;

BINGX_API_SECRET — API secret;

BINGX_TESTNET — true/false для тестового або реального підключення;

TRADING_MODE — simulation або live;

TELEGRAM_BOT_TOKEN — токен бота;

TELEGRAM_CHAT_ID — канал/група/чат для дублювання повідомлень;

SYMBOL — торговий інструмент;

TIMEFRAME — таймфрейм;

FIRST_SELL_QTY_COIN — стартовий розмір у coin;

USE_EQUITY_PCT_BASE — режим розрахунку стартового розміру;

EQUITY_FOR_SIZING_USDT — equity для розрахунку;

TP_PERCENT — цільовий TP;

CALLBACK_PERCENT — trailing callback;

MARGIN_CALL_LIMIT — максимальна кількість шортів;

MAX_ORDERS_PER_3MIN — ліміт ордерів за 3 хвилини;

USE_HIGH_LOW_TOUCH — логіка touch по high/low або body.

Нефункціональні вимоги
Підтримка відновлення стану після перезапуску.

Захист від дублювання ордерів.

Повний audit trail у логах.

Просте перемикання між simulation і live.

Чітка структура конфігурації без хардкоду секретів.

Усі параметри мають бути документовані в .env коментарями.

Критерії приймання
Бот відтворює логіку Pine Script у Python без втрати ключових правил стратегії.

Є два режими: simulation і real trading.

Кожен ордер пишеться в локальний лог і дублюється в Telegram.

Є робоча підтримка BingX API.

Є .env з коментарями до кожного параметра.

Стан стратегії коректно відновлюється після рестарту.
